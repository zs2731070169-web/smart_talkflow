
> **目标**:让平台在「编排执行中崩溃」后,**自动检测存活、自动恢复重跑**,使流程不因进程崩溃而永久卡死,实现「崩了自己爬起来」的 agent 基本能力——同时保证多实例并发安全、可重复执行幂等。
>
> **定位**:本文是《传统业务系统接入 Agent 落地计划》在**可靠性 / 自恢复维度**的专项补充。原计划与已落地的「Saga 补偿 + 幂等状态机」解决了「业务失败」的一致性,但未覆盖「**平台进程崩溃**」(OOM / 重启 / kill)导致的 `process` 永久卡 `running` 这一**基础设施级**留洞。
>
> **核心原则**:判崩溃靠**信号(心跳)**而非**时间(started_at)**;恢复靠**整笔回滚 + 重做**(对齐已有 Saga 补偿);多实例安全靠**数据库原子条件更新**(不引入 redis 锁的 TTL/竞态复杂度)。

## 1. 问题背景

### 1.1 崩溃留洞:`process` 卡 `running` 锁死幂等

平台侧 DB **没有贯穿执行的大事务**(每个 `db_session` 独立 commit,因执行中要发 HTTP、不能裹在 DB 事务里)。`IdempotencyChecker.check()` 插入 `status=running` 占位后 `commit`,之后才执行 workflow。若执行中进程崩溃:

| 崩溃点 | 后果 |
|---|---|
| `check` commit 之后、`execute` 之前 | `process` 永远 `running`,无 `_finalize` |
| `execute` 中途(某步 adapter 调用后) | `process` 永远 `running`,`process_step` 半截 |
| `_finalize` 之前 | `process` 永远 `running` |

**锁死幂等**:同一 `business_key` 再次请求 → 命中 `running` → `_check_process` 返回 `Action.REJECT`(拒绝并发重入)→ **用户无法重试、流程永久卡死**。

```
请求2 ──> dispatch ──> IdempotencyChecker.check(business_key)
                          │ 命中 status=running
                          └──> REJECT「流程进行中,拒绝执行」  ← 用户被卡死的旧实例挡住
```

且此时 yudao 侧真实状态未知(可能 submit 已成功、approve 未跑),平台侧留洞不可观测。

### 1.2 为什么不能「纯 `started_at` + N 分钟超时判崩溃」

若用 `started_at` 超时判崩溃,会**误杀长任务**:未来走真实 BPM 人工审批、或下游慢调用的流程可能跑几小时/一两天,纯时间阈值分不清「真崩了」和「真在跑(只是慢/在等)」。

### 1.3 agent 自恢复是基本能力

一个编排 agent 不能因为一次崩溃就永久锁死、等人手动清库——「**崩了自己爬起来重跑**」是智能编排的基本要求。人工兜底只能作为重试耗尽后的最后手段,不能是常态。

------

## 2. 设计目标与原则

| 目标 | 手段 |
|---|---|
| 检测存活 | **心跳**:`heartbeat_at`,执行中持续续,崩溃则停;判崩溃依据「**无心跳超时**」(非 `started_at`) |
| 自动恢复 | 崩溃 → **补偿旧残留(cancel)→ 从头重跑**(整笔回滚 + 重做,复用已有 Saga 补偿) |
| 多实例并发安全 | **数据库原子条件 UPDATE** 防重复 recover,无需 redis 锁 |
| 可重复执行幂等 | `WHERE status='running'` 条件更新,重复执行对已终态的 0 行影响 |
| 精简 | 不引入 redis 锁 / fencing token / 对账等(后续按需) |

**原则**:判崩溃靠信号不靠时间;恢复对齐已有 Saga(补偿 + 幂等);多实例安全下沉到 DB 原子操作。

------

## 3. 总体方案

### 3.1 崩溃检测:心跳

`process` 加 `heartbeat_at`。执行中(`run_steps` 每步)调 `renew_heartbeat` 续期;进程崩溃则停止续期。

```
存活 = heartbeat_at 持续更新(执行中每步续)
崩溃 = heartbeat_at 长期无更新(进程没了)
判据 = status='running' AND heartbeat_at < now - timeout   ← 无心跳超时,而非 started_at 超时
```

长任务只要在跑(每步续)就不会被判崩;只有进程真没了(停续)才超时。

### 3.2 崩溃恢复:补偿旧残留 + 从头重跑

```
watchdog 周期扫描
   │
   ▼
detect_stuck(timeout) ──> 命中「running 且无心跳超时」的 process
   │
   ▼
对每个 stuck process:
   ① DB 原子条件 UPDATE 抢 recover 权(heartbeat_at=now, attempt+1)── 多实例只一个成功
   ② attempt > max_retry? ── 是 ──> 标 failed(放弃,交人工)
   ③ 否:重建 operator(快照)+ workflow + inputs
   ④ 补偿旧残留:process_step.completed 的 bookingId ──> cancel yudao 旧预订
   ⑤ 清旧 process_step + 重置 process(running)
   ⑥ run_steps 从头重跑
   ⑦ 补偿失败 ──> 标 failed(交人工/对账)
```

语义等价于「整笔回滚 + 重做」:把崩溃前已对 yudao 产生的副作用(cancel 掉),再从头跑一遍。

## 4. 硬约束:yudao 非幂等

### 4.1 直接重跑会撞时段冲突

yudao `submit` 用 `insertOrUpdate` 且平台不传 id(每次新建),`validateTimeConflict` 会拒同时段重复预订。**崩溃前若 submit 已成功(yudao 已建预订),直接从头重跑 submit → 撞时段冲突 → 重跑失败**。

### 4.2 因此必须「重跑前补偿旧残留」→ 必须持久化 bookingId

要 cancel 旧残留,得知道 bookingId。但崩溃后内存 `ctx.results` 没了。**bookingId 只能从持久化恢复**:

- `process_step` 表本就有 `external_ref` 列,但之前去掉 `external_ref_key` 后**不再写入**。
- 恢复:`workflow_engine._exec_step` 的 `finish_step` 把 `AdapterResult.data`(会议室即 bookingId)写入 `process_step.external_ref`。
- recover 时查 `process_step.completed` 的 `external_ref` → cancel 对应 yudao 预订。

**这不是过度设计,是 yudao 非幂等前提下自动重跑绕不开的前提。**

### 4.3 重跑要代签 yudao → 必须存 operator 快照

重跑重新发 yudao 请求,需要代签头(`X-Operator-Userid` / `tenant-id` / HMAC),即重建 `OperatorContext`。崩溃后原请求的 operator(roles/tenant)只在内存,故:

- `process` 加 `operator_snapshot JSON`(user_id / roles / tenant);
- `dispatcher` 入口 check 时把 operator 传入存;
- recover 时从快照重建 `OperatorContext`。

------

## 5. 多实例防重

多实例水平扩展时,每个实例的 watchdog 都会扫到同一批 stuck process。防重复 recover **不用 redis 锁**,改用 **DB 原子条件 UPDATE**:

```sql
UPDATE process
SET heartbeat_at = NOW(), attempt = attempt + 1
WHERE id = ? AND status = 'running' AND heartbeat_at < <阈值>
```

- **原子性**:条件 UPDATE 是原子的,只有一个实例能影响 1 行(抢到 recover 权);其它实例再 UPDATE 时 `heartbeat_at` 已被刷新 → 条件不满足 → 0 行 → 跳过。
- **无 TTL / 无崩溃竞态**:相比 `SET NX EX` 的 redis 锁(锁 TTL 要 > recover 时长、持锁崩溃导致锁过期被误抢),DB 条件更新更稳。
- **可重复执行幂等**:`WHERE status='running'`,重复执行对已 `failed`/已恢复的 0 行影响。

> redis 分布式锁可作为补偿/重跑期间的并发「双保险」,但**主防重靠 DB UPDATE**,MVP 不引入 redis 锁。

## 6. 工程落地

### 6.1 心跳(检测存活)

| 文件 | 改动 |
|---|---|
| `db/smart_talkflow_init.sql` | `process` 加 `heartbeat_at DATETIME NULL`(放 `finished_at` 后);改 SQL 需 `docker compose down -v && up -d` 清卷重建 |

| `src/orchestrator/workflow_engine.py` | `run_steps` 每步 `_exec_step` 前 `await renew_heartbeat(process_id)` |

### 6.2 bookingId 持久化(补偿前提)

| 文件 | 改动 |
|---|---|
| `src/orchestrator/workflow_engine.py` | `_exec_step` 的 `finish_step` 写 `external_ref=resp.result.data`(会议室即 bookingId) |
| `src/infra/step_recorder.py` | `finish_step` 已支持 `external_ref` 参数(表本就有该列),由 workflow_engine 传入即可 |

### 6.3 operator 快照(重跑代签)

| 文件 | 改动 |
|---|---|
| `db/smart_talkflow_init.sql` + `models.py` | `process` 加 `operator_snapshot JSON NULL` |
| `src/infra/idempotency.py` | `IdempotencyCheckRequest` 加 `operator_snapshot` 字段;`check()` 存入 process |
| `src/orchestrator/dispatcher.py` | `_check_idempotency` 把入口 operator 序列化为 snapshot 传入 |

### 6.4 自动重跑(`src/orchestrator/recovery.py` 新)

`async def recover_stuck(max_retry: int) -> int`:
1. `stuck = await detect_stuck(settings.process_heartbeat_timeout)`。
2. 对每个 stuck process:
   - **DB 原子条件 UPDATE 抢权**(`status='running' AND heartbeat_at<阈值` → `heartbeat_at=now, attempt+1`);0 行 = 被其它实例抢或已变终态 → 跳过。
   - `attempt > max_retry` → 标 `failed`(放弃)。
   - 否则:重建 operator(snapshot)→ `registry.get_workflow(process.process_key)` → inputs(`process.input_params`)。
   - 补偿旧残留:`process_step.completed` 的 `external_ref`(bookingId)→ 对应 `workflow.steps` 的 `compensate`(cancel yudao)。
   - 清旧 `process_step` + 重置 `process`(running,清旧 result)。
   - `set_request_context`(operator/trace/process_id)→ `workflow_engine.run_steps(process_id, steps)`(从头)。
   - 补偿失败 → 标 `failed`。
3. 返回恢复条数。

> recovery 需访问 `WorkflowRegistry` + `workflow_engine`,故放在 `orchestrator/`(编排层),`infra/` 只提供 `detect_stuck` / `renew_heartbeat` 等数据访问。

### 6.5 后台 watchdog

| 文件 | 改动 |
|---|---|
| `src/infra/heartbeat.py`(新) | `heartbeat_watchdog()`:`while True: await sleep(interval); await recovery.recover_stuck(max_retry)`,异常 `except Exception` 不退出 |
| `src/main.py` | `lifespan` 启动 `task = asyncio.create_task(heartbeat_watchdog())`,停机 `task.cancel()` + `await gather(task, return_exceptions=True)` |

### 6.6 配置(`src/conf/config.py`)

```python
process_heartbeat_timeout: int = 300   # 无心跳超时秒(判崩溃阈值)
process_recovery_interval: int = 60    # watchdog 扫描间隔秒
process_max_retry: int = 3             # 自动重跑上限(超限标 failed 交人工)
```

均有默认值,不进 `_required_non_blank`。

## 7. 重试计数与放弃

`process.context.attempt`(幂等层已有该字段)记录重跑次数。每次 recover 抢权 `attempt+1`:

- `attempt <= max_retry`:补偿旧残留 + 从头重跑。
- `attempt > max_retry`:标 `failed`(error 含「重跑次数超限」),**交人工/对账**——避免对同一不可恢复任务无限重跑(如 yudao 持续冲突、补偿持续失败)。

这是「自动恢复」与「人工兜底」的边界:能自愈的尽量自愈,反复失败则停下交人。

## 9. 边界与未覆盖(后续阶段)

| 项 | 说明 | 后续 |
|---|---|---|
| 断点续跑(从失败步) | 当前 = 整笔回滚 + 从头重跑;不从崩溃点续 | 若重跑代价大,再做步级续跑 |
| 对账(查 yudao 真实状态) | 补偿失败 / 留痕不一致时,以 yudao 为真相源对齐 | 独立最终防线 |
| redis 分布式锁 | DB UPDATE 已防多实例重复 recover;锁仅作补偿期间双保险 | 多实例 + 补偿并发高时再加 |
| fencing token | attempt 计数 + DB 条件更新已防「旧实例复活写」的常见情况 | 极端竞态再加 |
| BPM 长审批 | 长等待应拆成 `waiting_external` 状态(回调驱动),不占 `running` | 接真实 BPM 时 |

## 10. 演进路线

1. **本计划**:心跳检测 + 自动重跑(补偿旧残留 + 从头重跑)+ DB UPDATE 多实例防重 + 重试计数。
2. **后续**:对账任务(查 yudao 对齐平台真实状态,补偿/重跑失败的最终兜底)。
3. **再后续**:断点续跑(步级恢复,减少重跑代价);`waiting_external` 状态 + 回调驱动(支撑 BPM 长审批等真正长任务)。

------

> **与已落地能力的关系**:本计划的「补偿旧残留」复用已落地的 **Saga 补偿**(`workflow_engine._compensate` + `cancel_booking`);「重试计数」复用幂等层的 `attempt`;「幂等防重」复用 `process` 的 `UNIQUE(process_key, business_key)`。新增的核心是**心跳存活检测**与**recovery 调度**,以及为支撑自动重跑而恢复的 `external_ref` 持久化与 `operator_snapshot`。
